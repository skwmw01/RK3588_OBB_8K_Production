# 03 · 方案原理 & 消融实验

## 🧠 核心原理

### 输入输出总览

```
输入: 8192 × 8192 × 3 (uint8) 航拍大图
   │
   ├─→ 按 stride=1848 切 5×5=25 个 2048×2048 patch
   │
   ├─→ 每个 patch resize 到 1024×1024 (LINEAR)
   │
   ├─→ 塞进 3 个 NPU 核（每核 8-9 个）
   │   每核自己跑：crop → resize → RGB → NPU 推理 → OBB decode
   │
   ├─→ 所有核 join 后合并 dets（平均 407 raw dets）
   │
   └─→ Rotated NMS (IoU=0.3) → 最终 42-47 个 det

输出: List[(cx, cy, w, h, angle, conf, cls)]
      cx,cy,w,h  都是原图 8192 坐标系
      angle      弧度，[-π/4, 3π/4]
      conf       0-1
      cls        0-14 (DOTA 15 类)
```

### 为什么是这些数字？

#### 2048 patch size

| 候选 | 优点 | 缺点 |
|---|---|---|
| 512 | 单 tile 快 | tile 数多（256+），总时间长 |
| 1024 | 模型原生尺寸 | 8K 需要 81 tile |
| **2048** | **5×5=25 tile，空间效率高** | 需要 resize（仅 1.5ms） |
| 4096 | 4 tile 超少 | 8K→4K 信息丢失严重 |

#### 5×5 = 25 tile 的意义

- 25/3 = 8 余 1：每核扛 8-9 tile，**最均衡的静态均分**
- 覆盖率 100%（overlap 200 保证边界不漏）
- 网格对称，没有"粗扫盲区"

#### overlap 200 px

- 太小（50）：边界目标被切成两半
- 太大（500）：tile 数膨胀
- **200** 对于 100-300 px 航拍目标来说，保证目标至少能完整落在一个 tile

#### 为什么 resize 到 1024 而非直接用 @2048

**不 resize 直接 @2048 模型**：
- 3 核并发扩展率仅 1.85×（带宽争用）
- 25 tile × 120ms = 3000+ ms ❌

**直接切 1024 不 resize**：
- 覆盖 8K 需要 81 tile
- 81/3 × 22 ≈ 594 ms 推理，加 decode+pre 超 1s ❌

**切 2048 resize 1024 @1024 模型**：
- 保留 2× 空间视野
- 用带宽友好的 @1024 模型
- 25 tile 刚好 1s 内 ✅

## 🧪 完整消融实验

### 实验 1：扫描策略对比

| 策略 | tile 数 | 时间 | 召回 | 结论 |
|---|---|---|---|---|
| 全扫 @1024 (GT 基准) | 81 | 4552 ms | 100% | baseline |
| 全扫 @2048 | 25×2048 | 3241 ms | 51.1% | @2048 带宽瓶颈 |
| 全扫 @1024 stride 匹 2048 rule | 100×1024 | 3179 ms | 46.8% | 反而更慢+掉召回 |
| Hot-expand v1 (8 邻) | 55 | 2819 ms | 59.6% | 超 2× 预算 |
| Hot-expand + 十字 4 邻 | ~43 | 2167 ms | 59.6% | 超 2× |
| Hot + 限 25 tile | 25 动态 | 918 ms | 51.1% | 勉强 1s |
| Hot + 限 15 tile | 15 | 609 ms | 51.1% | 最快但召回低 |
| **✅ 2048→1024 规整 5×5** | **25** | **815 ms** | **63.8%** | **👑 最优** |

### 实验 2：Resize 算法对比

| 算法 | 时间 | 召回 | 说明 |
|---|---|---|---|
| NEAREST | 843 ms | 59.6% | 最快但锯齿 |
| **LINEAR** | **827 ms** | **57.4%** | ✅ 速度+质量平衡 |
| AREA | 861 ms | 57.4% | 对缩小常用但无优势 |
| CUBIC | 908 ms | 61.7% | 召回高但 +80ms |

**LINEAR 胜出**。注：此实验 conf=0.2，conf=0.15 下 LINEAR 召回 63.8%。

### 实验 3：置信度阈值扫描

| conf | 中位时间 | 召回 | raw dets | final |
|---|---|---|---|---|
| **0.15** | **815 ms** | **63.8%** | 407 | 42-47 |
| 0.20 | 843 ms | 57.4% | 392 | 38 |
| 0.25 | 845 ms | 57.4% | 371 | 35 |
| 0.30 | 837 ms | 57.4% | 355 | 35 |

**conf=0.15 最优**：召回 +6.4%，时间不变。再低（0.10）只是冗余框变多。

### 实验 4：核数扩展性

@1024 模型：

| 核数 | 时间 | 加速比 |
|---|---|---|
| 1 核 | 2802 ms | 1.0× |
| 2 核 | 1224 ms | 2.29× |
| **3 核** | **839 ms** | **3.34×（超线性！）** |

对比 @2048 模型：

| 核数 @2048 | 时间 | 加速比 |
|---|---|---|
| 1 核 | 5550 ms | 1.0× |
| 3 核 | 3240 ms | 1.71× ❌ |

**@1024 扩展率 111%，@2048 仅 57%。差 2 倍 → 这就是选 @1024 的根本原因。**

### 实验 5：输入尺寸对比

| 输入 | 时间 | 召回 |
|---|---|---|
| 8000×8000 (先 resize) | 759 ms | 57.4% |
| **8192×8192 (原图)** | **798 ms** | **57.4%** |

直接用 8192 原图省 40ms。切片规则照样 5×5（最后一行/列右靠边界）。

### 实验 6：并发策略对比

| 方案 | 时间 | 说明 |
|---|---|---|
| Queue 轮询，decode 串行 | 1421 ms | ❌ decode 800ms 成瓶颈 |
| 静态均分 pre+inf，decode 串行 | 1421 ms | ❌ decode 未并行 |
| **静态均分，全塞线程** | **815 ms** | ✅ decode 也并行 |

**关键：pre/infer/decode 三件事都要塞进工作线程，只有 NMS 留主线程。**

### 实验 7：Decode 耗时（隐藏瓶颈）

| conf | raw dets | decode 累计（3 核各自） |
|---|---|---|
| 0.15 | 407 | 263-313 ms |
| 0.20 | 392 | 280-304 ms |
| 0.25 | 371 | 280-330 ms |
| 0.30 | 355 | 257-284 ms |

Decode 是**第二大瓶颈（35%）**，目前用 Python numpy 实现。

**未来优化**：C 扩展 / Cython 重写可砍 60%，挪到 RKNN 模型内可砍 90%。

## 📉 为什么召回只有 63.8%？

### 理论上限

全扫 @1024 × 81 tile 定义为 GT（47 目标，召回 100%）。

### 损失分布（估计）

| 损失来源 | 估计丢失 | 原因 |
|---|---|---|
| Resize 2048→1024 信息损失 | ~6 个 | <50px 小目标变 <25px 难检 |
| 切片边界截断 | ~3 个 | overlap 200 对大目标不够 |
| INT8 量化精度 | ~2 个 | INT8 vs FP16 轻微掉 |
| DOTA 预训练域不匹配 | ~6 个 | 8K 航拍 vs DOTA 训练分辨率 |

### 提高召回的路线（保持 1s）

| 方案 | 预期召回 | 代价 |
|---|---|---|
| 切 1024 resize 768 → @768 模型 | +2-3% | 模型要重量化 |
| 2048→1024 用 CUBIC | +2-4% | 时间 +80ms（可能超 1s） |
| QAT 微调 @1024 adapter | +10-15% | 训练 1-2 天 |
| 重训 yolov8s-obb + 剪枝 | +15-20% | 工作量大，风险高 |

### 放弃 1s 限制

| 方案 | 召回 | 时间 |
|---|---|---|
| 2048→1024 + CUBIC | 65-67% | 920 ms |
| 81 tile @1024 全扫 | 95-100% | 4552 ms |
| 81 tile + conf=0.1 | 100% | 4552 ms（GT）|

## 💡 核心设计决策总结

| 决策 | 选项 | 最终 | 理由 |
|---|---|---|---|
| 切片尺寸 | 1024/2048/4096 | **2048** | 5×5=25 恰好 3 核均分 |
| Overlap | 50/100/200/500 | **200** | 大目标完整覆盖 |
| 推理尺寸 | 1024/2048 | **1024** | 带宽友好，3 核线性扩展 |
| Resize 算法 | NEAREST/LINEAR/AREA/CUBIC | **LINEAR** | 速度+质量平衡 |
| conf 阈值 | 0.1/0.15/0.2/0.25/0.3 | **0.15** | 召回天花板，时间不变 |
| NMS IoU | 0.3/0.5/0.7 | **0.3** | 旋转框标准 |
| 核数 | 1/2/3 | **3** | 超线性扩展 |
| 模型格式 | FP16/INT8 | **INT8** | 3× 速度，精度无损 |
| ONNX 版本 | 官方/airockchip | **airockchip** | 官方 INT8 塌方 |
| 并发 | queue/静态 | **静态均分** | 少 8% overhead |
| 输入 | 8000/8192 | **8192** | 省一次 resize |
